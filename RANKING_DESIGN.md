# 거지주차.com 랭킹 & 포인트 시스템 설계

> 작성일: 2026-04-19
> 상태: 설계 확정 / 구현 대기

## 1. 목적

단순히 제보 수로 경쟁하는 게 아니라, **양질의 정보를 꾸준히 공유한 사용자**를 가시화하고 보상하기 위한 시스템. 누적 포인트에 따라 커뮤니티 내 차등 혜택을 제공해 자정 기능과 참여 동기를 동시에 높인다.

---

## 2. 포인트 공식 (확정)

사용자 점수는 **실시간 계산** 방식으로 가져간다. 별도로 `users` 시트에 `points` 칼럼을 두지 않는다. `get_ranking` API가 호출될 때마다 `mainSheet`와 `comments` 시트를 순회해 점수를 합산한다. 이유: 후기/제보가 삭제될 때 포인트 롤백 로직을 따로 관리하지 않아도 항상 정합성이 보장된다. 사용자 수천 명 수준까지 Apps Script로 충분히 처리 가능.

### 점수 산정

- 주차 정보 제보 1건 등록: **+5점 (기본 점수)**
- 내 제보에 타인이 남긴 후기 1건당: **별점 값 그대로 적립** (1★=+1, 2★=+2, ..., 5★=+5)
- 본인이 본인 제보에 단 후기는 점수에서 제외 (어뷰징 방지, 아래 3번 참고)

### 계산 예시

A씨가 제보 3건을 올렸고, 각 제보에 다음과 같이 후기를 받음:

- 제보1: 5★ 2개, 4★ 1개 → 10 + 4 = 14점
- 제보2: 5★ 1개 → 5점
- 제보3: 후기 없음 → 0점

A씨 총점 = 제보 기본점수 (5 × 3 = 15) + 받은 후기 점수 (14 + 5 + 0 = 19) = **34점**

## 3. 어뷰징 방지 (확정)

점수 시스템이 도입되는 순간 어뷰징(부정 행위)이 최대 리스크다. 최소한 아래 두 가지는 시스템 오픈과 동시에 반드시 적용한다.

### 3.1 자기 제보에 자기 후기 금지 (MUST)

백엔드 `add_comment` 핸들러에서 후기 작성자(`user`)와 해당 제보의 제보자가 같으면 거절한다. 에러 메시지: "본인이 올린 제보에는 후기를 남길 수 없습니다."

구현: `comments` 시트에 insert 전, `mainSheet`에서 해당 제보(target_id 기준)의 제보자 아이디를 찾아 비교.

### 3.2 기존 "아이디당 1후기" 규칙 유지

이미 구현됨. 부계정으로 같은 제보에 여러 번 별점 박는 걸 제한.

### 3.3 추후 고려 (초기 버전에서는 미적용)

- 후기 개수 임계치: "후기 1개 이하인 제보는 점수 반영 제외" → 오픈 초기엔 제보 자체가 적을 수 있으므로 미적용. 어뷰징 패턴 감지되면 도입.
- 제보당 점수 상한: "한 제보당 최대 50점까지만 인정" → 마찬가지로 초기엔 미적용.
- IP 기반 다중 계정 제한: 구현 비용 대비 효과 낮음. 당장은 생략.

## 4. 혜택 단계 (확정)

**하이브리드 방식 — 단계별 권한 부여 + 삭제는 신고 중심**. 강력한 직접 삭제권은 당분간 주지 않음. 부작용이 크기 때문.

### 포인트 구간별 혜택

| 포인트 | 혜택 | 비고 |
|---|---|---|
| 50점 | 이름 옆 ⭐ 뱃지 | 명예 혜택, 기능 변화 없음 |
| 200점 | 내 후기·댓글 수정 권한 | 지금은 삭제 후 재작성만 가능 |
| 500점 | 수다방 글·댓글 신고 버튼 활성화 | 500점 이하 유저는 신고 불가 |
| 1000점 | 내 제보의 후기 정렬 우선권 + 🏆 뱃지 | 인포윈도우 상단에 노출 |
| 2000점 | 신고 가중치 2배 | 1회 신고가 2표로 집계 |
| 5000점 | (차후 검토) 직접 삭제권 + 로그 | 당분간 보류 |

### 삭제 정책 (신고 기반)

- 일반 유저: 삭제 권한 없음
- 500점 이상: 글/댓글 신고 가능
- 같은 글/댓글에 **3명 이상 신고** 누적 → 자동으로 "숨김 처리"
- 숨김 처리된 글/댓글은 일반 유저 목록에서 안 보임 (DB에는 남아있음)
- 추후 관리자(쌍칼 계정)만 복구·영구삭제 가능

→ 이 방식은 한 명의 악의적 행위자가 글을 지울 수 없고, 여러 고신뢰 유저의 합의가 있어야만 삭제가 일어남.

## 5. 랭킹 페이지 UI (확정)

현재 "🏆 랭킹" 버튼은 "준비 중" 상태. 이 버튼을 활성화한다.

### 표시 항목

- 페이지 상단: 상위 10명 (닉네임 + 점수 + 뱃지)
- 페이지 하단: 내 순위와 내 점수 (로그인한 닉네임 기준)
- 내 순위 옆: 다음 혜택까지 남은 점수 ("200점까지 OO점 남음")

### 정렬

- 1차: 총점 내림차순
- 2차 (동점 시): 제보 수 내림차순
- 3차 (그래도 동점 시): 최초 제보 일시 오름차순 (먼저 시작한 사람 우선)

### 영구 누적 방식

시즌제(월 리셋)가 아닌 **영구 누적 방식**으로 유지. 단순하고 꾸준한 기여자에게 유리. 추후 "이달의 별" 서브 랭킹을 별도로 추가할 수 있음(차후 검토).

## 6. 기술적 구현 계획 (단계별)

큰 기능이라 한 번에 배포하지 않고 **4단계로 나눠** 구현·검증한다. 각 단계 완료 후 실제로 동작하는지 확인하고 다음으로 넘어간다.

### 단계 1: 어뷰징 방지 (최우선)

- Code.gs `add_comment` 에 자기 제보 후기 차단 로직 추가
- 검증: 본인 계정으로 본인 제보에 후기 시도 → 에러 반환 확인

### 단계 2: 포인트 계산 & 랭킹 API

- Code.gs에 `get_ranking` 핸들러 신설
- 입력: 없음 (또는 옵션으로 `top=10`)
- 출력: `[{ rank, user, score, reportCount, receivedReviews, avgRating }, ...]`
- 내부 로직: mainSheet 순회 → 유저별 제보 수 집계 → commentSheet 순회 → 유저별 받은 별점 집계 → 합산 → 정렬

### 단계 3: 랭킹 페이지 UI

- index.html: "🏆 랭킹" 버튼에서 `openRanking()` 호출
- script.js에 `openRanking()`, `renderRanking()` 함수 신설
- 디자인: 수다방과 일관된 스타일 (노란 테마)
- 내 순위·점수 표시

### 단계 4: 혜택 단계 적용

- 가장 가벼운 것부터: 50점 ⭐ 뱃지 (닉네임 옆 표시)
- 200점 후기·댓글 수정 권한 (신규 `edit_comment`, `edit_board_comment` API)
- 500점 신고 권한 + `board_reports`, `board_comment_reports` 시트 신설
- 신고 3회 누적 시 `handleBoardFetch`에서 해당 글 제외
- 1000점 🏆 뱃지 + 후기 정렬 우선권
- 2000점 신고 가중치 2배

→ **각 혜택마다 백엔드와 프론트엔드를 함께 수정**해야 하므로 한 번에 하나씩.

## 7. 데이터 스키마 변경 요약

### 기존 시트 (변경 없음)

- `mainSheet`: 제보 정보
- `comments`: 주차 후기
- `board`: 수다방 글
- `board_comments`: 수다방 댓글
- `users`: 아이디·비번

### 신규 시트 (단계 4-3 반영 · 확정 스키마)

- `board_reports`: [게시글ID, 신고자, 신고자가중치, 신고일]
- `board_comment_reports`: [게시글ID, 댓글작성자, 댓글일시(ISO), 신고자, 신고자가중치, 신고일]

신고 시점의 가중치를 함께 저장한다. 나중에 신고자의 점수가 올라가도 과거 신고에 소급 적용하지 않음(예측 가능성·성능).

**포인트 자체는 별도 저장하지 않음** — 매번 계산.

## 8. 결정 이력 (Why)

- **누적 합계 vs 평균 별점**: 누적 합계 선택. 이해 쉽고, 꾸준히 활동한 사람 자연스럽게 우대.
- **실시간 계산 vs 저장**: 실시간 계산 선택. 데이터 정합성 보장 쉬움. 규모 커지면 캐싱으로 최적화.
- **직접 삭제권 vs 신고제**: 신고제 선택. 소규모 커뮤니티에서 직접 삭제권은 악용 리스크가 너무 큼. 다수 합의 방식이 안전.
- **영구 누적 vs 시즌제**: 영구 누적 선택. 구현 단순. 신규 유저 재미는 "이달의 별" 서브 랭킹으로 차후 보완.

## 9. 추후 검토 과제

- 악성 계정 패턴 감지 (같은 비번 사용, 짧은 시간 내 다량 계정 생성)
- "이달의 별" 월간 서브 랭킹
- 제보 게시 후 N일 이상 유지돼야 점수 인정 (긴급 삭제 후 재등록 어뷰징 방지)
- 5000점 직접 삭제권 도입 여부 재검토
- 관리자(쌍칼) 전용 대시보드 (숨김 처리된 글 복원·영구 삭제)
- 유저 제재 시스템 (어뷰징 확인 시 점수 깎기 / 권한 회수)
